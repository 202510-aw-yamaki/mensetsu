# インシデント対応

この記録は、`usage-device` と `audio-camera-desktop` の修正で起きた手戻りを、後任が同じ形で繰り返さないために残す。
要点だけでなく、初動・修正・悪化・再把握の順で、何をした結果どう崩れたかまで残す。

## 対象

- `pre-interview-checks/usage-device.html`
- `pre-interview-checks/audio-camera-desktop.html`
- `css/pre-interview-checks-detail.css`
- `js/usage-device-layout.js`
- `js/audio-camera-layout.js`
- `重要.md`
- `作業.md/index.md`
- `作業.md/pre-interview-checks.md`
- `作業.md/usage-device.md`
- `作業.md/audio-camera-desktop.md`

## 何が起きたか

PC幅で `slide-body` を 750px 基準に寄せる修正を入れた際、`usage-device.html` と `audio-camera-desktop.html` を同じ粒度で扱ってしまった。
その結果、`usage-device` 側の「画像は必ず全部見せる」「アスペクト比は固定する」という意図と、`audio-camera` 側の「可変してよいが全体は見せる」という意図を混同した。

`contain` / `cover` / `fill` の役割と、JS が担っている余白制御の役割を分けずに触ったため、意図と違う見え方になった。

## 初動

1. `slide-body` を PC幅で 750px に固定する方向へ変更した。
2. その時点で `usage-device` と `audio-camera` を別ページとして分けて確認せず、共通の高さ問題として扱った。
3. `CSS` と `JS` のかかり方、さらに `重要.md` と作業メモの設計意図を、先に全体で確認しなかった。

この時点で、後任が取るべきだった確認順は崩れていた。

## 修正

1. 画像が余白を持たないようにする意図で、`usage-device-visual` と `audio-camera-visual` に同じ方向の高さ制御を入れた。
2. 画像の全体表示を守るべきページと、可変でもよいページの差を十分に分けずに触った。
3. `usage-device` と `audio-camera` を共通症状としてまとめ、ページ固有の差を分けなかった。

この段階では、「何を直すか」だけを見て、「どこがどう変わるか」の確認が不足していた。

## 悪化

1. `usage-device` では灰色背景が見える状態になった。
2. `audio-camera` では本来の全体表示方針からずれた。
3. さらに、JS が担っていた余白調整の役割まで曖昧になり、どの層が何を制御しているか分かりづらくなった。
4. 修正を重ねるほど、`CSS` の上塗りと `JS` の上塗りが混ざり、どの記述が本来の責務なのか追いづらくなった。

つまり、問題は1回の誤修正だけではなく、誤修正を前提に追加修正を重ねたことで、構成の見通しまで悪化したことにある。

## 再把握

1. `重要.md` と `作業.md` 群を再確認した。
2. そこで、`usage-device` と `audio-camera` は同列ではないと整理し直した。
3. `usage-device` は「全部見える・比率固定・余白を上へ逃がす」方針で扱う必要があると確定した。
4. `audio-camera` は「可変でも全体表示」の方針で扱う必要があると確定した。
5. 同じ `pre-interview-checks-detail.css` を使っていても、ページごとの役割は別に読む必要があると再確認した。

## 全体把握

- `css/common.css` は全ページ共通の土台で、ページ枠・ヘッダー・フッター・共通カードの基本形を持っている。
- `css/pre-interview-checks-detail.css` は第1章の詳細ページ共通だが、`usage-device` と `audio-camera` は同じファイル内でも役割が違う。
- `js/usage-device-layout.js` は `usage-device` の右カラム余白を調整するためのもので、上の画像へ余白を逃がす役割がある。
- `js/audio-camera-layout.js` は `audio-camera` の左カラム高さと右カラムの余白配分を調整するためのもので、画像の見え方を維持しながら高さを合わせる役割がある。
- `作業.md` の各記録には、`usage-device` と `audio-camera` を分けて扱うべきだった経緯がすでに残っている。

## この記録で残すべき教訓

### 1. `usage-device.html`

- 画像は必ず全部見せる。
- アスペクト比は固定する。
- `class="image-panel detail-visual usage-device-visual"` の可変で生じる余白は、枠の中で上へ逃がす。
- 具体的には、右カラムの上の画像の上部に余白を寄せ、下側に灰色の余白が残りにくい形へ戻す。
- `JS` は、余白を均等に散らすのではなく、上側へ寄せるために使う。
- `CSS` は、画像の切り抜きではなく、全体表示を前提にしたまま高さ配分だけを受け持つ。

### 2. `audio-camera-desktop.html`

- `class="image-panel detail-visual audio-camera-visual"` は可変のままでよい。
- アスペクト比が多少変わってもかまわない。
- ただし、画像の全体は常に見せる。
- `JS` は、画像の比率を守るためではなく、可変の箱の中で上下の余白と高さを整えるために使う。
- `CSS` は、`fill` を含む全体表示寄りの扱いを維持し、切り抜きは避ける。

## 時系列

### 初動

- `slide-body` を PC幅で 750px に固定する方向へ変更した。
- その時点で `usage-device` と `audio-camera` を別ページとして分けて確認せず、共通の高さ問題として扱った。
- `CSS` と `JS` のかかり方を確認せずに、まず修正に入った。
- `重要.md` と作業メモ群を先に読み切らず、既存の設計意図の確認が不足した。

### 修正

- 画像が余白を持たないようにする意図で、`usage-device-visual` と `audio-camera-visual` に同じ方向の高さ制御を入れた。
- 画像の全体表示を守るべきページと、可変でもよいページの差を十分に分けずに触った。
- `usage-device` と `audio-camera` を共通症状としてまとめ、ページ固有の差を分けなかった。
- 修正の結果がどう変わるかを先に示さず、作業を進めた。

### 悪化

- `usage-device` では灰色背景が見える状態になった。
- `audio-camera` では本来の全体表示方針からずれた。
- さらに、JS が担っていた余白調整の役割まで曖昧になり、どの層が何を制御しているか分かりづらくなった。
- 修正を重ねるほど、`CSS` の上塗りと `JS` の上塗りが混ざり、構成が冗長になった。
- 結果として、どの記述が共通土台で、どの記述がページ固有なのか追いづらくなった。

### 再把握

- `重要.md` と `作業.md` 群を再確認し、`usage-device` と `audio-camera` は同列ではないと整理し直した。
- そのうえで、`usage-device` は「全部見える・比率固定・余白を上へ逃がす」、`audio-camera` は「可変でも全体表示」の別方針として扱う必要があると確定した。
- この再把握を経てから、修正案を出す段階に戻すべきだと判断した。

## 期待する変化

- `usage-device.html` では、画像の全体表示と比率維持が保たれる。
- `usage-device.html` の余白は、画像の中で分散させず、上側に寄せられる。
- `audio-camera-desktop.html` では、可変の中でも画像全体が見え続ける。
- 両ページを同じ粒度で扱う誤りをやめ、ページ固有の意図を壊さない。
- `作業.md` と `重要.md` の記録と矛盾しない形で修正できる。

## 進め方

- まず `CSS` / `JS` / `作業.md` / `重要.md` を再確認する。
- 次に `usage-device` と `audio-camera` を別々に扱う。
- 変更は、必要最小限のセレクタとスクリプトに限定する。
- 修正前に、どのファイルのどの挙動がどう変わるかを先に提示する。
- この提案に対する確認が取れてから着手する。

## 再発防止

### プロジェクト全体

- 「直せそう」だけで着手しない。
- まず `重要.md`、`作業.md`、共通 `CSS`、共通 `JS`、ページ別 `CSS`、ページ別 `JS` の順で責務を確認する。
- 変更前に、どのレイヤーで直すべきかを決める。`共通土台` で直すのか、`ページ固有` で直すのかを分ける。
- 1つの修正で `CSS` と `JS` を同時に上書きしない。必要な層だけを触る。
- 既存の設計意図を壊す可能性がある場合は、作業前に修正案を出して確認を取る。
- 修正後は、関連ページも含めて影響範囲を確認する。単一ページだけ見て完了にしない。
- 作業メモには、結果だけでなく「なぜその修正が必要だったか」「どの層を触ったか」「何を触らなかったか」も残す。
- 後任が読んだときに、同じ誤りを起こさない粒度で残す。

### ページ個別

- `usage-device` は「全部見える・比率固定・余白は上へ逃がす」を維持する。
- `audio-camera` は「可変でも全体表示」を維持する。
- 似たレイアウトでも、画像の扱いと余白の扱いを同じものとして扱わない。
- 画像の切り抜き、背景の見え方、JS の余白補正は別々の責務として考える。
